Database Notifications — What scope?

PostgreSQL LISTEN/NOTIFY — how deep should we go?

A

Listener Only

Subscribe to channels and see events in a live log. That's it. Simple, focused, useful for debugging triggers and pub/sub patterns.

  • Subscribe/unsubscribe to channels
  • Live event log with timestamps
  • Filter/search events
  • Copy event payload
  • Clear log
B

Listen + Send

Everything in A, plus ability to send NOTIFY from the UI. Useful for testing notification-driven workflows end-to-end.

  • Everything in Listener Only, plus:
  • Send NOTIFY with channel + payload
  • Quick-send recent payloads
  • JSON payload editor
C

Full Pub/Sub Dashboard

Dedicated panel with channel management, event history persistence, payload inspection, and stats (events/sec, avg payload size). More of a monitoring tool.

  • Everything in Listen + Send, plus:
  • Persistent event history (stored locally)
  • Channel stats (events/sec, avg size)
  • Auto-reconnect on connection drop
  • Multiple channel tabs
