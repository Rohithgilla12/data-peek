## Design Context

### Users
Developers (backend engineers, full-stack devs, data-curious builders) who need to quickly peek at database contents, debug queries, and explore schemas. They're typically mid-task — checking a migration, verifying data, or writing a query — and want to get in, get answers, and get out. They value speed and keyboard efficiency over visual polish.

### Brand Personality
**Fast. Honest. Modern devtool.**

- **Fast**: Sub-2-second launch, instant query results, no loading spinners where avoidable. The UI should feel like it responds before you finish thinking.
- **Honest**: No bloat, no telemetry, no dark patterns. Honor system licensing. What you see is what you get.
- **Modern devtool**: Feels like it belongs alongside Linear, Raycast, and Warp — not alongside DBeaver or pgAdmin. Clean, opinionated, keyboard-first.

### Emotional Goals
- **Confidence**: Users should feel in control of their data at all times
- **Calm focus**: The interface recedes — data and queries are the protagonist
- **Speed**: Every interaction should feel instant and effortless

### Aesthetic Direction

**Visual tone**: Minimal, technical, monospace-native. The UI is a tool, not a destination. Dense where useful, spacious where it aids scanning.

**References**:
- **Linear** — Command palette, keyboard-first, clean dark mode, information density without clutter
- **Raycast** — Speed-first interaction model, minimal chrome, results-focused

**Anti-references**:
- **DBeaver / pgAdmin** — Cluttered toolbars, nested dialogs, enterprise-heavy chrome, overwhelming options
- Anything with wizard flows, tab explosions, or gratuitous icons

**Theme**: Dark and light mode, system-preference aware. Dark mode is the primary design target.

### Color System

**Canonical accent**: OKLCH blue (hue 250°). The desktop app is the source of truth — marketing site and docs should align to this.

- Light: `oklch(0.45 0.15 250)` — deep, confident blue
- Dark: `oklch(0.65 0.15 250)` — lighter blue for dark backgrounds
- Blue was chosen over cyan for its familiarity and readability across both themes

**Palette summary**:
| Role | Light | Dark |
|------|-------|------|
| Background | `oklch(0.985 0 0)` | `oklch(0.16 0.005 260)` |
| Foreground | `oklch(0.14 0 0)` | `oklch(0.92 0 0)` |
| Primary accent | OKLCH blue (hue 250°) | OKLCH blue (hue 250°) |
| Destructive | Red-orange | Red-orange |
| Muted | Subtle blue tint | Subtle blue tint |

### Typography
- **Sans**: Geist Variable — UI labels, navigation, buttons
- **Mono**: Geist Mono Variable — queries, data, code, and the marketing site (monospace-first identity)
- The desktop app uses mono as the base body font, reinforcing the technical identity

### Design Principles

1. **Data is the UI**: Query results, schemas, and data grids are the primary interface. Everything else (chrome, navigation, controls) should minimize its visual footprint.

2. **Speed over ceremony**: No confirmation dialogs where undo works. No wizard flows. Prefer inline editing, command palettes, and keyboard shortcuts over modal workflows.

3. **Dense but scannable**: Embrace information density (like a terminal), but use spacing, typography weight, and subtle color to create hierarchy. Never sacrifice scannability for compactness.

4. **Native, not web-feeling**: The app should feel like a native macOS/Windows/Linux tool — proper titlebar behavior, system font rendering, platform-appropriate shortcuts, no "website in a window" energy.

5. **Quiet confidence**: No flashy animations, no gradients in the app chrome, no decorative elements. Let the data speak. Use motion only for state transitions (expand/collapse, tab switching) and keep it under 200ms.
