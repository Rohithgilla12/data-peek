01

Plaintext credential storage

● Critical Effort M Risk Med main/storage.ts · index.ts
▶

Root cause

Connections — including DB password, sshConfig.password, passphrase, and privateKeyPath — are persisted through the unencrypted DpStorage facade at index.ts:39, landing as cleartext JSON in userData/data-peek-connections.json. The encrypted DpSecureStorage exists but is disabled (storage.ts:170), and getEncryptionKey() falls back to a hardcoded static key 'data-peek-fallback-key-v1' (storage.ts:39, 71). Same plaintext store also holds AI keys and license data.

Why it "corrupts": on Linux, Electron's safeStorage backend can change between launches, so decryptString throws → the catch at storage.ts:50 deletes the key file and generates a new random key → electron-store can no longer decrypt the existing JSON, which reads as corruption. The static-fallback ↔ real-key mismatch causes the same. The "fix" they shipped (disable encryption) was a band-aid for an unstable key, not a flaw in encryption itself.

Fix approach

  1. Make the key stable and fail-closed — never substitute a constant. If safeStorage is unavailable or the key can't persist, refuse to write secrets (in-memory only) rather than encrypt with a constant or fall back to plaintext.
  2. Stop deleting the key file on a single read failure; only regenerate when genuinely absent.
  3. Pin the safeStorage backend on Linux early in startup.
  4. Re-enable DpSecureStorage for connections under a new store name + one-time migration from the plaintext file, then delete the plaintext file.
— before · storage.ts getEncryptionKey (abridged) 
if (!safeStorage.isEncryptionAvailable()) {
  log.warn('safeStorage not available, using static fallback key')
  cachedEncryptionKey = 'data-peek-fallback-key-v1'   // ← constant key
  return cachedEncryptionKey
}
try {
  if (existsSync(keyFilePath)) {
    cachedEncryptionKey = safeStorage.decryptString(readFileSync(keyFilePath))
    return cachedEncryptionKey
  }
} catch {
  unlinkSync(keyFilePath)   // ← destroys the key on a transient failure
}
+ after
if (!safeStorage.isEncryptionAvailable()) {
  throw new EncryptionUnavailableError('OS secure storage (safeStorage) is unavailable')
}
if (existsSync(keyFilePath)) {
  // Do NOT delete on failure — a transient backend mismatch must not destroy the key
  cachedEncryptionKey = safeStorage.decryptString(readFileSync(keyFilePath))
  return cachedEncryptionKey
}
const newKey = randomBytes(32).toString('hex')
writeFileSync(keyFilePath, safeStorage.encryptString(newKey)) // throws if it can't persist — fail closed
cachedEncryptionKey = newKey
return cachedEncryptionKey
+ after · index.ts — pin backend, migrate, use secure store 
if (process.platform === 'linux') safeStorage.setUsePlainTextEncryption(false)

store = await DpSecureStorage.create<{ connections: ConnectionConfig[] }>({
  name: 'data-peek-connections-secure',
  defaults: { connections: [] }
})
await migratePlaintextConnections(store) // reads legacy json once, then unlinks it
!
Migration concern: use a new store name so a fresh encrypted file is created — never feed an encrypted blob to the old plaintext reader. Migration must be idempotent (guard on existsSync + empty target) and delete the plaintext file only after a successful write. Linux users whose key already churned re-enter credentials once. Apply the same treatment to AI keys / license data in a follow-up.

Verify

  • Add a connection, quit, inspect data-peek-connections-secure.json → values are ciphertext; old data-peek-connections.json is gone.
  • Relaunch repeatedly (esp. Linux/CI without keyring) → no key-file deletion, no "recreating" logs, connections persist.
  • Force isEncryptionAvailable() false → app refuses to persist secrets, never writes plaintext.
07

No renderer logging strategy · no redaction

● Medium Effort M Risk Low ~145 console.* sites
▶

Root cause

Main has a redacting logger (main/lib/logger.ts), but the renderer has no logger and ~125 raw console.* calls (main has ~20 more). Confirmed leak: query-editor.tsx:62 logs the entire activeConnection object — password and SSH secrets included — to DevTools. This directly compounds Issue 1.

Fix approach

  1. Extract SENSITIVE_KEYS + redactSensitive into a shared module so main and renderer share one policy. Extend the key list to catch passphrase, privateKeyPath, sslKey, connectionString (substring match misses these today).
  2. Add a renderer logger that mirrors main and redacts objects before printing.
  3. Migrate console.* → logger; fix the query-editor.tsx:62 leak immediately.
  4. Add an ESLint no-console rule with the logger module as the only exception.
+ renderer/src/lib/logger.ts
import { redactSensitive } from '@shared/redact'

function format(args: unknown[]): unknown[] {
  return args.map((a) => (typeof a === 'object' && a !== null ? redactSensitive(a) : a))
}

export function createLogger(module: string) {
  const prefix = `[${module}]`
  return {
    debug: (...a: unknown[]) => import.meta.env.DEV && console.debug(prefix, ...format(a)),
    info: (...a: unknown[]) => console.info(prefix, ...format(a)),
    warn: (...a: unknown[]) => console.warn(prefix, ...format(a)),
    error: (...a: unknown[]) => console.error(prefix, ...format(a))
  }
}

+ after
const log = createLogger('query-editor')
log.debug('handleRunQuery called', { isExecuting, hasConnection: !!activeConnection })

Electron renderer sandbox disabled

● Medium Effort S Risk Low window-manager.ts:57
▶

Root cause

window-manager.ts:57 sets { preload, sandbox: false } and does not pin contextIsolation / nodeIntegration. The agent audited the preload (preload/index.ts) and its one dep @electron-toolkit/preload: nothing requires sandbox:false — only contextBridge, ipcRenderer, and process.* are used, all available under the sandbox. So it can be flipped safely.

— before
webPreferences: {
  preload: join(__dirname, '../preload/index.js'),
  sandbox: false
}
+ after
webPreferences: {
  preload: join(__dirname, '../preload/index.js'),
  sandbox: true,
  contextIsolation: true,
  nodeIntegration: false,
  webSecurity: true
}

Keep the preload limited to electron/IPC so a future Node built-in at preload scope doesn't break the sandbox. The existing setWindowOpenHandler (deny + openExternal) is unaffected.

Verify

  • Launch app → window.api / window.electron still populate (run a query, list connections).
  • In the renderer console, typeof require and Node globals are undefined (isolation enforced).
  • No "Insecure webPreferences" warning in the dev console.

05

Silent-failure / catch-and-default pattern

● High Effort S Risk Low–Med 6 production sites
▶

Root cause & breadth

Of ~345 catch blocks, the swallowing pattern narrows to 6 production sites. The IPC boundary is already correct — IpcResponse<T> + wrapHandler exist and db:schemas even falls back to stale cache with a refreshError flag. The real risk is internal helpers that catch and never throw.

Site Returns Bucket
mysql-adapter.ts:1505 getLocks [] must-surface — hides perms failure as "no locks", no log
data-generator.ts:244 resolveFK [] must-surface — emits bad data with null FKs silently
scheduler-service.ts:614 / dashboard-service.ts:704 [] recoverable but unlogged (bare catch {})

Convention

Reuse the existing boundary contract — do not add a new Result<T>. Helpers/adapters throw; the IPC handler wraps. A catch may return a default only if the default is semantically correct and it logs with context. Bare catch {} is banned.

— before · mysql-adapter.ts:1504
} catch {
  return []   // "no locks" — indistinguishable from failure
}
+ after
} catch (err) {
  log.error('Failed to fetch MySQL locks', err)
  throw err   // db:locks handler already wraps → surfaces as an error toast
}

  • Connect a MySQL user lacking performance_schema → Locks view shows an error, not an empty list.

Unvalidated casts on driver result rows

● High Effort S–M Risk Low zod already a dep do after #5
▶

Root cause

~378 as casts (zero as any). Two classes: bulk data-row casts (as Record<string, unknown>[] — acceptable, fields already coerced defensively) and introspection-shape casts that assert specific fields with no runtime check — these need validation. Confirmed: sqlite-adapter.ts:282-298 casting PRAGMA output, mssql-adapter.ts:275 column metadata. zod is already installed (^3.25.42).

  cid: number; name: string; type: string; notnull: number; dflt_value: string | null; pk: number

+ after
const PragmaTableInfo = z.object({
  cid: z.number(), name: z.string(), type: z.string(),
